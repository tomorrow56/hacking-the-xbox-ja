# 翻訳にもCI/CDを──『Hacking the Xbox』日本語版で試していること（高須正和）

最近、Andrew “bunnie” Huangの『Hacking the Xbox』を日本語に翻訳し、GitHub Pagesで公開する作業をしている。原著と同じCreative Commonsライセンスで、翻訳途中のものも含めて[GitHub](https://github.com/Nico-Tech-Shenzhen/hacking-the-xbox-ja)に置いている。

『Hacking the Xbox』は2003年に出版された、初代Xboxのセキュリティを解析していく過程についての本だ。単なる技術解説ではなく、当時MITの学生だったbunnieが、実際のハードウェアを手に取り、仮説を立て、失敗し、また別の方法を試しながら仕組みに迫っていく過程が書かれている。僕は以前、同じbunnieの『The Hardware Hacker』を日本語に翻訳したことがあり、今回も日本語で読めるようにしたいと思っていた。

とはいえ、300ページ近くある技術書を一人で翻訳するのは大変だ。今回はClaude CodeやChatGPTなど複数のAIを使っている。AIがなければ、少なくとも今のタイミングで始めようとは思わなかっただろう。

実際に始めてみると、AIが本を翻訳してくれるという単純な話ではなかった。むしろ、AIが大量の作業をできるようになったからこそ、仕様を作り、成果物をレビューし、問題を仕組みに戻して改善するエンジニアリングが重要になる。今回やっていることも、翻訳というよりかなり普通のエンジニアリングに近い。

今回やっているのは、最初に完璧な翻訳ルールを作り、そのルールをAIに守らせる方法ではない。まずAIに文章を訳させ、僕がそれを読んで手で直す。次に、その修正からChatGPTにルールを抽出させ、新しいルールで章全体を翻訳し直す。この繰り返しで、翻訳と翻訳方法の両方を更新している。

## 機械翻訳を手で直すところから始めた

『Hacking the Xbox』の原文をAIに渡せば、数分で日本語が出てくる。文章としてだいたい読めるが、違和感があり、読んでいて疲れる。bunnieの文章はこうではないはずだ。

章全体を通読すると、よりおかしさに気づく。技術用語の訳が少しずつ違ったり、英文の構造を引きずった読みづらい日本語になっていたりする。日本の読者向けに、英語圏にしかない言葉を出す時は説明を足しているのだが、それをAIが余計に解釈して、原文にはない説明や見出しを付け加えることもある。

一番大きな問題は、bunnieが自分でXboxを解析しながら考えている感じがなくなることだった。「僕はこう考えた」「こう試した」「これは失敗だった」という個人の体験が、AIを通すと一般的な技術解説のようになる。情報としてはだいたい合っていても、『Hacking the Xbox』を読む面白さが減っている。

そこで、最初は機械翻訳を僕が読んで、おかしい部分を手で直していった。一人称の「私」を「僕」に直す。技術用語を、その分野で使われる表現に変える。意味が分からない文章をChatGPTにきいて、原文と再参照しながら意味が分かるようにする。英語の一文をそのまま長い日本語の一文にせず、読みやすい単位に組み直す。

この段階では、翻訳ガイドが先にあったわけではない。実際の訳文を読んで、「これは違う」と思った部分を直すところから始まっている。

人間が先に行ったのは、抽象的な仕様の設計ではなく、成果物を見て違和感を持つことだった。

## 人間が違和感を見つけ、AIがルールにする

手で修正すると、その箇所はよくなる。でも、別の章を機械翻訳すると、同じ問題がまた出てくる。一人称を直し、技術用語を直し、余計な説明を削る作業を、章ごとに繰り返すことになる。

そこで、原文、機械翻訳、僕が手で直した文章と、なぜそう直したかをChatGPTに渡し、複数の箇所に共通する問題を抽出させることにした。個別の修正を並べるのではなく、今後の翻訳にも使えるルールにしてもらう。

たとえば、「私」を「僕」に変えるという修正は、単なる単語の置換ではない。bunnieが自分の試行錯誤を語る文章を、後から説明する教科書のような文体にしない、というルールになる。原文にない補足を削るという修正も、特定の一文の問題ではなく、AIが読者に親切にしようとして内容を追加しない、というルールにできる。

僕が行っているのは、成果物を読んで「ここが変だ」「これは面白くない」と判断することだ。ChatGPTは、その具体的な修正を比較し、ほかの箇所にも適用できる形に抽象化する。出てきたルールを僕が確認し、必要なら直したうえで、[TRANSLATION_GUIDE.md](https://github.com/Nico-Tech-Shenzhen/hacking-the-xbox-ja/blob/main/TRANSLATION_GUIDE.md)や[glossary.tsv](https://github.com/Nico-Tech-Shenzhen/hacking-the-xbox-ja/blob/main/glossary.tsv)に追加する。

つまり、人間が感覚を出し、AIが抽象化する。最終的にそのルールを採用するかどうかは、また人間が判断する。

Claude Codeには、更新された翻訳ガイドと用語集を読ませ、同じ問題が章内のほかの場所にないかを調べさせる。必要なら新しいルールを使って章全体を翻訳し直し、サイトをビルドする。

[第8章](/ja/ch08)で頻出するFPGAを使った信号処理の部分は、僕自身がその分野への関心が薄く、面白いか、勉強になるか、読む気になるかの判断がつかなかった。そこでFPGAに造詣の深い[@susutawari](https://x.com/susutawari)さんに読んでもらい、[Issue](https://github.com/Nico-Tech-Shenzhen/hacking-the-xbox-ja/issues)を立ててもらった。

その指摘をもとに僕が文章を直し、ChatGPTに同じ傾向の問題を章全体から抽出させ、今後の章にも適用できるルールにした。そのルールを使って、第8章全体をClaude Codeで再翻訳している。

第8章で見つかった問題は、技術的な表現が多い第5章、第6章、第7章、第9章、第10章、第11章にもある可能性がある。新しいルールは、これから翻訳する章だけでなく、すでに翻訳した章の見直しにも使える。

一か所で感じた違和感を、その一文の修正で終わらせず、過去と未来の翻訳を改善する仕様に変える。今回の作業で一番面白いのは、この部分だ。

## 翻訳にもCI/CDを持ち込む

CI/CDは、もともとはソフトウェア開発で使われる言葉だ。CIはContinuous Integration、継続的インテグレーションの略で、小さな変更を頻繁に共通のリポジトリへ統合し、そのたびにビルドやテストを行う。CDはContinuous DeliveryまたはContinuous Deploymentの略で、検証を通った成果物を、いつでも公開できる状態にしたり、実際に継続して公開したりする仕組みを指す。

大きな変更を最後にまとめて確認するのではなく、小さな変更を加えるたびに検証し、問題があれば修正する。変更、テスト、公開、フィードバックの周期を短くして、成果物を少しずつ改善していく。

今回の翻訳で繰り返しているのは、次のような作業だ。

1. 原文を機械翻訳する。
2. 人間が章全体を読み、誤訳、技術的な問題、日本語の違和感を手で直す。
3. 修正前と修正後をChatGPTに渡し、共通する問題をルールとして抽象化し、
4. 人間がルールをレビューし、翻訳ガイドや用語集を更新する。
5. ChatGPTがそれをもとにClaude Codeへの指示を生成
5. Claude Codeが新しいルールを追加。　ルール追加時に、矛盾するルールの解決や、重複する指示の抽象化などもvalidationとして組み込む(Skill化)
6. 上のプロセスを何度か繰り返し、ある程度ルールがupdateされたら、章ごとに再翻訳
7. GitHub Pagesで公開し、読者からの指摘を次の修正とルール作成に使う。

今回の翻訳はソフトウェアではないし、日本語として面白いかどうかを自動テストで判断できるわけでもない。それでも、成果物を一度作って完成とせず、レビューで見つかった問題を仕様に戻し、更新した仕様で成果物を再生成する流れはCI/CDに近い。

GitHubには翻訳結果だけでなく、翻訳ルールと、その変更履歴も残る。文章を一つずつ直すのではなく、次に作る文章がよくなる仕組みを直している。

これは、本の内容を変えるという話ではない。たとえば今回の訳では、セガや任天堂などの日本メーカーや製品はカタカナや日本語で表記し、Microsoftは英語表記のままにしている。もともとすべて英語で文章を書いたbunnieの頭の中にはない判断であり、日本語で読む人がどう受け取るかを考えて、翻訳者が決めなければならないことだ。

入力があり、成果物があり、レビューがあり、そこで見つかった問題から仕様を作り、もう一度実装する。今回やっていることは、翻訳というより普通のエンジニアリングに近い。

## AI時代には、仕様とレビューがもっと重要になる

今回、最初に仕様を作ってから翻訳したわけではない。先に成果物を作り、人間が読んで直し、具体的な修正から仕様を作った。実際の成果物を見なければ、何をルールにするべきか分からなかったからだ。

AIが実装の多くを行うようになっても、最初から完全な仕様を作ることはできない。出てきたものを見て、「この指示では足りない」「この修正は、ほかの場所にも適用するべきだ」と判断し、その結果から仕様を更新する必要がある。

そのとき、人間が行う仕事は二つある。一つは、成果物を見て違和感を持つこと。もう一つは、AIが抽象化した仕様が正しいかをレビューすることだ。

詳しくなければ、出てきたもののどこがおかしいのか分からない。AIが作ったルールが問題を正しく一般化しているのかも判断できない。AIが実装や抽象化を手伝ってくれるほど、人間には成果物と仕様の両方をレビューする能力が必要になる。

ただし、仕様をきちんと作れば全部解決するわけでもない。誤字がない、用語が統一されている、原文の情報が抜けていない、余計な説明を追加していない、といった条件はルールにできる。でも、完成した文章を読んで面白いかどうかは別だ。

AIに「面白い文章にしてください」と指示することはできる。でも、出てきた文章が本当に面白いのか、この本の面白さを残しているのかは、最終的には人間が判断するしかない。成果物に意味があるのか、世の中に出す価値があるのか、そもそもなぜこれを作るのかも、人間が決めることだ。

正しいけどつまらない文章はある。読みやすくても、原著者の声がしない文章もある。その本を翻訳する意味が残っているかどうかは、仕様への適合だけでは判断できない。

## 「AIに作らせればいい」という話が嫌いだ

AIについて、「これからは人間が詳しくなくても、AIに聞けばなんとかなる」「知識を覚えたり、自分で考えたりする必要がなくなる」という話をよく見る。これまで教育や学習にたいして関心があるようには見えなかった人が、AIが出てきた途端に「もう勉強しなくていい」「教科書はAIに作らせればいいじゃん」と言い出すことも多い。

僕はこの考え方が嫌いだし、単に態度として好きではないというだけでなく、間違っていると思っている。

もちろん、AIによって、これまで専門家にしかできなかったことを多くの人が始められるようになるのはよいことだ。英語が得意でなくても海外の情報を読めるし、プログラミングを始めたばかりの人でも動くコードを作れる。僕自身、この翻訳でAIの能力に頼っている。

でも、作り始められることと、よいものを作れることは違う。新しいものを作る仕事は、その対象が好きでないとなかなかよいものにならない。

ここでいう「好き」は、仕事を楽しくするための気分や、プロフィールに書く趣味のことではない。気になって仕方がない、もっと知りたい、誰もやっていないなら自分で試したい、うまくいかないと悔しい、できたものをほかの人にも見せたい、という欲望に近いものだ。

食欲や性欲とまったく同じだと言いたいわけではないけど、役に立つかどうかも分からないものを調べ、頼まれていないものを作り、効率が悪くても気になるから続ける性質は、生物としての人間に組み込まれているのだと思う。人類がこれまでやっていなかった変わったことを始め、できることの範囲を広げてきたのも、そういう性質によるものだろう。

AIは生物ではないので、この欲望を持っていない。与えられた目的に対して、限られた計算資源の中でもっともらしい出力を返すように作られている。頼まれてもいないのに、面白そうだから寝る時間を削って余計なものを作ったり、役に立つか分からないけど気になるから何年も続けたりはしない。

何を作るのか、なぜ作るのか、どこが面白いのか、何を失ったら作る意味がなくなるのかは、AIから自動的に出てくるものではない。人間が対象を見て、考えて、決める必要がある。

今回の翻訳で具体的に何をしたかというと、AIに本を丸ごと渡して、日本語にしてもらったわけではない。AIが作った翻訳を読み、どこが間違っているかだけでなく、どこが面白くなくなっているかを考え、その理由を翻訳の仕様へ戻し、もう一度作らせる仕組みを作った。

最終的に、成果物に意味があるか、世の中に出す価値があるか、何を失ったら作る意味がなくなるかを判断するのは人間だ。そして、その人間は誰でもよいわけではない。

翻訳なら、その本を好きな人が必要だ。ソフトウェアなら、そのソフトウェアが解決しようとしている問題を気にしている人、製品なら、それによって起きる変化に興味がある人が必要になる。詳しいだけでなく、成果物を面白がって読み、「ここが大事だ」「こうなったら意味がない」と言える人でなければならない。FPGAの話で [@susutawari](https://x.com/Susutawari) さんにメンション飛ばして読んでもらったのも、彼がFPGAに詳しいことはもちろんだが、それだけではなくて、「この文章を面白がって読んでくれそうだな」という期待があった。面白がって読んでくれないとダメなのだ。

人間とAIの大きな違いは、計算能力や記憶容量だけではなく、この「これが面白いからやりたい」があるかどうかだと思う。そして、AIと一緒に仕事をするときに一番重要になるのも、人間の側にあるこの感情ではないかと思う。

僕が『Hacking the Xbox』の翻訳をレビューできるのも、英語と日本語が分かるからだけではない。僕自身がハードウェアハッキングやオープンハードウェアに長く関わり、この本を面白いと思っている。著者のbunnieを知り、『The Hardware Hacker』も日本語に翻訳した。この本がどういう人によって書かれ、どこが面白いのかについて、自分なりの理解があるし、「翻訳したいと思って、実際に手を動かした」から、こうなっている。

だから、情報としては正しく見えても、「これはbunnieの文章に見えない」と判断できる。もちろん、その判断が必ず正しいわけではないので、GitHubで公開し、ほかの読者からの[Issue](https://github.com/Nico-Tech-Shenzhen/hacking-the-xbox-ja/issues)も受け付けている。

## AIは人間の代わりに好きになってくれない

AIによって、文章を入力したり、コードを一行ずつ書いたりする作業は減っていくだろう。でも、人間が知らなくてよくなるわけでも、考えなくてよくなるわけでもない。

人間が考えることは、何を作るのか、なぜ作るのか、何を正しいとするのか、どういう仕様にするのか、出てきたものは面白いのか、それを世の中に出す意味があるのか、ということへ移っていく。AIが大量に作れるようになるほど、その判断をする仕事は増える。

AI時代だからエンジニアリングが不要になるのではない。AI時代だからこそ、仕様を作り、成果物をレビューし、問題を一般化し、仕組みに戻して改善するエンジニアリングが必要になる。

そして、その工程の始まりにも終わりにも、「これが面白いからやりたい」と思う人間が必要になる。AIは多くの作業を手伝ってくれるけど、人間の代わりにその成果物を好きになってはくれない。

『Hacking the Xbox』日本語版は、まだ翻訳途中だ。一度翻訳した章も、翻訳ガイドが更新されれば、また見直して再翻訳する。読んでいて誤訳や技術的におかしい部分、日本語として違和感がある部分を見つけたら、[GitHub Issue](https://github.com/Nico-Tech-Shenzhen/hacking-the-xbox-ja/issues)で教えてほしい。

その指摘は一つの文章を直すだけでなく、新しい翻訳ルールになり、これから翻訳する章や、すでに翻訳した章を改善するかもしれない。翻訳だけでなく、翻訳する仕組みそのものを継続的に改善していく。それが今回、『Hacking the Xbox』日本語版で試していることだ。

おそらくそれは、この本でbunnieが言っている、以下に近いのではないか。

Xboxハッキングコミュニティに何ができるかを考えるとき、急いで飛び込もうとしたり、夢中になれないプロジェクトや、自信を持って実行できないプロジェクトに無理に取り組もうとしたりしなくていい。自分の番が来たときには自然にわかる——プロジェクトが自分の名前を呼ぶような感覚があり、自然にハックせずにはいられなくなるだろう。[第13章](https://takasumasakazu.net/hacking-the-xbox-ja/ja/ch13.html)
